# prest 開発者ガイドラインとコーディング基準 (Rules & Development)

`prest` プロジェクトに貢献する開発者や、コアロジックを修正・維持するにあたって、厳格に遵守すべきアーキテクチャの境界、コーディング規約、およびテストの基準について定義します。

---

## 1. クレイト・アーキテクチャ (Crate Architecture)

コードの疎結合性と再利用性を極限まで高めるため、`prest` は同一コードベース内で「ライブラリ」と「バイナリ」の役割を明確に切り離しています。

- **ライブラリ・クレイト (`prest`):** 
  `src/lib.rs` 内に実装。データの変換、パース、コンパイル、およびファイルライターロジックなど、コアとなるビジネスロジックをすべてカプセル化します。CLI パーサ（`clap` 等）への依存を**完全に排除**し、純粋なプログラム用インターフェースのみを公開します。
- **バイナリ・クレイト (`prest-bin`):** 
  `src/main.rs` および `src/cmd/` モジュール内に実装。CLI 入力パラメータの解析、値のバリデーション、およびサブコマンドのハンドリングを担当します。
- **依存フローの制約:** 
  `src/cmd/` (バイナリ) から `prest` (ライブラリ) への依存のみを許可します。ライブラリクレイトは、バイナリクレイト (`src/cmd/`) 内のコードを参照したり、CLI のグローバル状態に依存したりしては**絶対になりません**。

---

## 2. 必須コーディング基準 (Mandatory Coding Standards)

### 10行ルール (The 10-Line Rule)
コードの明瞭性、テスト容易性、およびモジュール性を最大限担保するため、**関数、メソッド、ヘルパー、またはテストコードの1関数あたりの行数は 10行以下** とします。
- 関数が 10行を超える恐れがある場合、必ず単一責任の小さなヘルパー関数へと細分化（リファクタリング）してください。
- このルールは、関数の責務を明確にし、コードレビューの負荷を減らすための強力な制約です。

### 早期リターン（Early Returns / Flat Code）
深くネストされた `if-else` ブロックの記述を禁止します。ガード句（Guard Clauses）を用いた早期リターンを徹底し、コードのインデントを平坦（フラット）に保つことで、可読性と制御フローの直観性を維持します。

### ドキュメント要件
- **すべての関数・メソッド**（内部用プライベート関数含む）には、その目的と挙動を簡潔に説明するコメントを必ず記述してください。
- **すべての公開（`pub`）関数・メソッド**には、Rustdoc 標準ドキュメント形式（`///`）を使用し、さらに実際に動作検証可能なコード例を含めた `/// # Examples` セクション（doctest）を必ず設けてください。

## 3. 品質保証と密閉テスト (Quality Assurance & Hermetic Testing)

### ドックテスト (Doctests)
公開 API の Rustdoc ブロックに記述されたコード例は、テストスイートの一部として自動実行されます。新機能の追加や API の変更時には、必ず動作する doctest をセットにしてください。

### 統合テスト (Integration Tests - `tests/`)
データ変換の全仕様、例外的なエッジケース、および CLI ルーティングモードの挙動は、`tests/` ディレクトリ配下にブラックボックス統合テストとして配置します。

### テスト実行とビルド基準
すべてのプルリクエストおよびコミットは、以下の基準を満たさなければなりません。
1. コンパイラ警告（Compiler Warnings）およびリンターエラーが **完全にゼロ** であること。
2. 以下のコマンドを実行し、すべてのテストがパスすること。
   ```bash
   cargo test
   ```

### テスト副作用のクリーンアップ（Hermeticity）
テストの実行は完全に決定論的であり、ネットワークなどの外部依存がないオフライン環境で動作する必要があります。また、テスト中に生成された一時ファイルやディレクトリは、`tempfile` などの RAII ツールを用いて、テスト終了時に**必ず自動で完全消去**されるように実装してください。

## 4. Jujutsu (`jj`) によるバージョン管理ワークフロー

このプロジェクトでは、従来の Git に代わる先進的なバージョン管理ツール **Jujutsu (`jj`)** を採用しています。

### 変更管理手順
開発時には、Git ブランチの代わりに Jujutsu の「ブックマーク（bookmark）」を使用して作業を行います。

1. **機能追加やバグ修正用ブックマークの作成**:
   ```bash
   jj bookmark create feat-my-feature
   ```
2. **変更の自動追跡**:
   Jujutsu は作業ディレクトリの変更を自動で追跡（コミット）します。手動で毎回 `git add` や `git commit` を行う必要はありません。
3. **コミットメッセージ（リビジョン説明）の設定**:
   詳細かつ意味のあるメッセージをセットします。
   ```bash
   jj describe -m "feat: xxx の変換パターンを追加

- data/ の json 構造から自動推論する機能の強化
- src/lib.rs への変換ロジックの実装
- テストケースの追加"
   ```
4. **テストの実施**:
   ```bash
   cargo test
   ```
5. **ステータスおよび差分の確認**:
   ```bash
   jj status
   ```

### コミットの基本ルール
- **アトミック（原子性）コミット**: 1つのリビジョン（コミット）には、正確に1つの論理的な変更のみを含めてください。複数の異なるバグ修正や、ドキュメント修正、機能開発を同一コミットに混在させてはなりません。
- **詳細な記述**: 変更理由（WHY）、影響範囲、技術的アプローチについて、複数行にわたる丁寧なコミット説明を記述してください。1行のみの簡素なコミットメッセージは受け入れられません。

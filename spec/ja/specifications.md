# prest 技術仕様とデータ変換 (Specifications)

`prest` は、静的ホスティング環境で動作する高性能な API を生成するための、様々な高度なビルド仕様、変換ロジック、および開発者支援（DX）機能を備えています。

---

## 1. 構成ティア (Configuration Tiers)

`prest` は、開発の段階や必要とされる柔軟性に応じて、3つの構成ティアをサポートしています。

- **Tier 1: ゼロコンフィグ (Zero-Config)**: 設定ファイルが存在しない場合、`prest` は `./data` 内の構造から「設定より規約」の原則に従って自動的に API を推論・構築します。
- **Tier 2: 規約オーバーレイ (`_config.json`)**: 自動推論される API ツリーの構造を模した JSON 設定を用いて、必要な箇所にのみ特定のディレクティブを適用するパッチ（上書き）を行います。
  - **自動探索（Auto-Discovery）**: エンジンは設定ファイルを `_config.json`, `_prest.json`, `.config.json`, `.prest.json` の順で自動的に探索し、最初に見つかったものを適用します。
- **Tier 3: プログラマティック設定 (`api.config.js`)**: JavaScript / TypeScript を使用して複雑なデータ変換関数を直接記述します。

---

## 2. シリアライザー（Serializer）と配信レイアウト（Layout）仕様

`prest` のコンパイルシステムは、データの物理構造を決定する **「シリアライザー（Serializer）」** と、ファイルシステム上の配置を決定する **「配信レイアウト（Layout）」** という、直交する2つの概念に整理されて定義されます。

### A. 3つの物理シリアライザー（Serializer）
各シリアライザーは、インプットデータ（JSON）を指定された形式で物理ファイル化（シリアライズ）します。

1. **`json` シリアライザー (基本)**
   - **仕様**: プレーンな JSON テキストデータとして書き出します。
   - **拡張子**: `.json`
2. **`typescript` (または `js`) シリアライザー**
   - **仕様**: データを JSON ではなく、ESM（ES Modules）形式のコード（例: `export const data = [...]`）に変換して書き出します。
   - **拡張子**: `.ts` (または `.js`)
   - **用途**: Astro, Next.js, SvelteKit などの SSG ビルドプロセスで直接モジュールとして `import` するフロントエンド用途に最適。
3. **`sqlite` シリアライザー**
   - **仕様**: データを単一のバイナリリレーショナルデータベース形式（SQLite）にコンパイルして出力。単一のバイナリを出力するため、このシリアライザーに `layout` 設定は不要（適用されません）。
   - **拡張子**: `.db` (または `.sqlite`)
   - **用途**: クライアント側で SQL.js や WASM版 SQLite を稼働させ、静的ホスティング上で高度な SQL クエリをローカルメモリ上で処理させたいケースに最適。

### B. 3つの配信レイアウト（Layout）

`layout` は、静的サーバーでのエンドポイント解決（URL）と物理ファイルの配置の関係を制御します。

- **`index` レイアウト (デフォルト)**: すべてのエンドポイントを `/endpoint/index.[ext]` として出力。最も互換性が高く、すべての静的サーバーで安全にクリーン URL が機能します。
- **`file` レイアウト**: すべてのエンドポイントを `/endpoint`（拡張子なしファイル）として直接出力。
  - **スマート・フォールバック仕様**: 物理的なファイル・ディレクトリ衝突を回避するため、配下にサブパスを持つコレクションはコンパイル時に自動で `.../index.[ext]` 形式に置換（フォールバック）されます。
- **`extension` レイアウト**: すべてのエンドポイントに直接拡張子を付与し、`/endpoint.[ext]` として出力。100% の Web サーバー互換。

## 3. 規約オーバーレイスキーマ (Tier 2 Routing)

「設定より規約」の思想を厳格に守るため、`_config.json` での高度な設定は **オーバーレイスキーマ** を用います。すべてのルートを明示的に定義するのではなく、自動推論される API ツリーと同じ構造の JSON を記述します。

先頭に `$` が付くキーは、その階層のデータを加工する **ディレクティブ（指示）** として解釈されます。`$` が付かないキーは **サブパス** を定義します。

```json
{
  "api": {
    "activities": {
      // ルートの /api/activities コレクション全体に適用される
      "$filter": [
        { "field": "is_public", "op": "eq", "value": true }
      ]
    },
    "job-histories": {
      // 新しいサブパス /api/job-histories/current を定義
      "current": {
        "$filter": { "field": "to", "op": "eq", "value": "Present" }
      }
    },
    "profile": {
      "$aggregate": ["job-histories", "activities", "degrees", "skills"]
    }
  }
}
```

## 4. 高度なフィルタリング (`$filter`)

`$filter` ディレクティブは、コレクションから特定の要素を抽出します。型安全性、実装の容易さ、およびデバッグのしやすさを確保するため、フィルタは厳格な `field`, `op`, `value` のオブジェクト構造を採用します。

### サポートするオペレータ (`op`)

- `eq` (Equal): 一致する
- `neq` (Not Equal): 一致しない
- `gt` (Greater Than): より大きい
- `gte` (Greater Than or Equal): 以上
- `lt` (Less Than): より小さい
- `lte` (Less Than or Equal): 以下
- `contains`: （文字列・配列が）含む
- `exists`: フィールドが存在する（`value` は boolean）
- `regeq`: 正規表現に一致する
- `regneq`: 正規表現に一致しない

**例:**
```json
"$filter": [
  { "field": "age", "op": "gte", "value": 18 },
  { "field": "status", "op": "eq", "value": "active" }
]
```

## 5. データサニタイズ (`$pick` & `$omit`)

パスワードハッシュや内部メモなどの機密情報が公開 CDN に漏洩するのを防ぐため、API ツリーの任意のノードでフィルタリングを適用できます。

- **`$pick` (ホワイトリスト)**: 指定したキーのみを保持。
- **`$omit` (ブラックリスト)**: 指定したキーを確実に削除。

---

## 6. データ集約とバンドル設計 (`$aggregate`)

複数のエンドポイントやデータセット全体を単一のエンドポイントにマージする「バンドル」は、起動時にデータベース全体をロードするクライアント側アプリケーションに最適です。

`$aggregate` ディレクティブを使用することで、任意のパスで複数のコレクションを1つの配列にマージできます。

**例: `/api/db` へのバンドル構築**
```json
{
  "api": {
    "db": {
      "$aggregate": ["users", "profile", "papers"]
    }
  }
}
```

---

## 7. CORS とプラットフォームヘッダーの自動生成

API を静的 CDN から配信し、異なるオリジン（フロントエンドドメイン）からアクセスする際の CORS 制約を解決するため、ビルド時にサーバー設定ファイルを自動生成します。
- **Cloudflare Pages / Netlify**: `_headers` ファイル
- **Vercel**: `vercel.json` ファイル

---

## 8. TypeScript 型定義ファイルの自動生成 (`.d.ts`)

安全な型安全性を確保するため、`prest` はビルドプロセス中に JSON 構造を解析し、対応する TypeScript 型定義ファイル (`.d.ts`) を自動生成します。
- **出力先**: `dist/types/` (例: `profile.d.ts`, `users.d.ts`)

---

## 9. キャッシュ & インクリメンタルビルド

大量の JSON データを処理する際のビルドを最適化するため、差分ビルド機能を備えています。
- ビルド時に SHA-256 ハッシュを計算し、`.static-api-cache.json` に保存。
- 次回以降のビルド時、ハッシュを検証して変更のあったファイルのみをコンパイル。

---

## 10. メディアアセットのバンドルとパス書き換え

JSON 内でローカルアセットを相対パスで参照している場合、これを自動検出し、メディアファイルを `/dist/api/assets/` に自動コピーの上、JSON内のパスを公開相対 URL に自動書き換えします。

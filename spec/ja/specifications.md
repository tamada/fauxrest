# prest 技術仕様とデータ変換 (Specifications)

`prest` は、静的ホスティング環境で動作する高性能な API を生成するための、様々な高度なビルド仕様、変換ロジック、および開発者支援（DX）機能を備えています。

---

## 1. 構成ティア (Configuration Tiers)

`prest` は、開発の段階や必要とされる柔軟性に応じて、3つの構成ティアをサポートしています。

- **Tier 1: ゼロコンフィグ (Zero-Config)**: 
  設定ファイルが存在しない場合、`./data` 内の構造から自動的にデフォルトのシリアライザー（`json`）と配信レイアウト（`index`）で API を自動構築します。
- **Tier 2: 簡易ルーティングファイル (`routes.yml`)**: 
  ルーティングのカスタマイズ、パスの上書き、JSONPath による部分抽出などを簡易指定します。
- **Tier 3: プログラマティック設定 (`api.config.js`)**: 
  JavaScript / TypeScript を使用して複雑なデータ変換関数を記述できます。

---

## 2. コア REST 変換パターン (REST Transformation Patterns)

カスタムルーティングを使用する際、`prest` はインプットデータを以下の 4 つの構造変換パターンで処理します。

1. **単一オブジェクト (`type: single`)**
   単一のターゲット JSON オブジェクトを指定のパスにそのまま配置します。
2. **コレクション (`type: list`)**
   JSON 配列をそのまま一覧エンドポイントとしてコンパイルします。
3. **アイテム詳細 (`type: detail`)**
   入力された配列を走査し、動的セグメント（例: `:id`）で指定されたキーの値に基づき、それぞれのアイテムに対して一貫した物理ファイルを個別出力します。
4. **グルーピング (`type: group`)**
   指定されたフィールドの値に基づいて配列オブジェクトをグループ化し、それぞれのグループごとに分割されたリソース一覧を出力します。

---

## 3. シリアライザー（Serializer）と配信レイアウト（Layout）仕様

`prest` のコンパイルシステムは、データの物理構造を決定する **「シリアライザー（Serializer）」** と、ファイルシステム上の配置を決定する **「配信レイアウト（Layout）」** という、直交する2つの概念に整理されて定義されます。

### A. 3つの物理シリアライザー（Serializer）
各シリアライザーは、インプットデータ（JSON）を指定された形式で物理ファイル化（シリアライズ）します。

1. **`json` シリアライザー (基本)**
   - **仕様**: プレーンな JSON テキストデータとして書き出します。
   - **拡張子**: `.json`
2. **`typescript` (または `js`) シリアライザー**
   - **仕様**: データを JSON ではなく、ESM（ES Modules）形式のコード（例: `export const papers = [...]`）に変換して書き出します。
   - **拡張子**: `.ts` (または `.js`)
   - **用途**: Astro, Next.js, SvelteKit などの SSG ビルドプロセスで直接モジュールとして `import` するフロントエンド用途に最適。
3. **`sqlite` シリアライザー**
   - **仕様**: データを単一のバイナリリレーショナルデータベース形式（SQLite）にコンパイルして出力。単一のバイナリを出力するため、このシリアライザーに `layout` 設定は不要（適用されません）。
   - **拡張子**: `.db` (または `.sqlite`)
   - **用途**: クライアント側で SQL.js や WASM版 SQLite を稼働させ、静的ホスティング上で高度な SQL クエリ（テーブル結合、インデックス検索等）をローカルメモリ上で処理させたいケースに最適。

### B. 3つの配信レイアウト（Layout）
`layout` は、静的サーバーでのエンドポイント解決（URL）と物理ファイルの配置の関係を制御します。

- **`index` レイアウト (デフォルト)**: 
  すべてのエンドポイントを `/endpoint/index.[ext]` として出力。最も互換性が高く、すべての静的サーバーで安全にクリーン URL が機能します。
- **`file` レイアウト**: 
  すべてのエンドポイントを `/endpoint`（拡張子なしファイル）として直接出力。
  - **スマート・フォールバック仕様**: ファイルシステムにおいて「同一名でフォルダとファイルは共存できない（`/api/papers` というコレクションファイルと `/api/papers/` という個別詳細の格納ディレクトリが衝突する）」という物理的限界を回避するため、**配下にサブパス（個別リソース等）を持つコレクションは、コンパイル時に自動で `.../index.[ext]` 形式に置換（フォールバック）** されます。サブパスを持たないリソースはそのまま拡張子なしファイルで書き出します。
- **`extension` レイアウト**: 
  すべてのエンドポイントに直接拡張子を付与し、`/endpoint.[ext]` および `/endpoint/{id}.[ext]` として出力。100% の Web サーバー互換。

## 4. データ集約とバンドル設計 (Data Aggregation & Bundle Pattern)

複数のエンドポイントやデータベース全体を、ひとつの巨大な `/db.json`（または任意の単一パス）に統合する「**バンドル**」は、ジェネレータ自体の物理出力モードの切り替えではなく、**ルーティング・集約（Aggregation）の設定定義**によって解決・実現されます。

### 集約機能（`aggregate`）による実現
メタキー `$index: { "aggregate": [...] }` を用いることで、指定した複数のコレクションを1つの配列にマージしたエンドポイントを任意のパスに構成できます。

- **例: `/db.json` へのバンドル構築 (`routes.yml`)**
  ```yaml
  "/db.json":
    $index:
      aggregate:
        - "data/users.json"
        - "data/profile.json"
        - "data/papers.json"
  ```
  これにより、クライアントは起動時に `/db.json` を1回 fetch するだけでデータベース全体の必要なアセットをメモリ上に読み込むことが可能になります。

---

## 5. 高度な JSONPath 抽出 (Advanced Node Targeting)

巨大なモノリシックデータから特定の部分ツリーのみを取り出して API 化したい場合、標準の JSONPath クエリを用いてターゲットノードを直接指定できます。
```yaml
# db.json の中の activities 配列のみを抽出してエンドポイントにする
"/activities": "data/db.json$.activities"
```

---

## 6. データサニタイズ (`pick` & `omit`)

クライアントに不必要なデータ、またはパスワードハッシュやドラフト状態のフラグなどの機密情報が公開 CDN に漏洩するのを防ぐため、コンパイル時にフィルタリングを行うことができます。
- **`pick` (ホワイトリスト方式)**: 指定したキーのみを保持。
- **`omit` (ブラックリスト方式)**: 指定したキーを確実に削除。

---

## 7. CORS とプラットフォームヘッダーの自動生成

API を静的 CDN から配信し、異なるオリジン（フロントエンドドメイン）からアクセスする際の CORS（Cross-Origin Resource Sharing）制約を解決するため、ビルド時に以下のサーバー設定ファイルを自動生成します。
- **Cloudflare Pages / Netlify 向け (`_headers` ファイル)**
- **Vercel 向け (`vercel.json` ファイル)**

---

## 8. TypeScript 型定義ファイルの自動生成 (`.d.ts`)

フロントエンドアプリケーション開発における安全な型安全性を確保するため、`prest` はビルドプロセス中に JSON 構造を解析し、対応する TypeScript 型定義ファイル (`.d.ts`) を自動生成します。
- **出力先**: `dist/types/` (例: `profile.d.ts`, `users.d.ts`)

---

## 9. キャッシュ & インクリメンタルビルド

大量の JSON データを処理する際のビルドのボトルネックを解消するため、`prest` は差分ビルド（インクリメンタルコンパイル）機能を備えています。
- ビルド時に全入力ファイルの SHA-256 ハッシュを計算し、`.static-api-cache.json` に保存。

---

## 10. メディアアセットのバンドルとパス書き換え

JSON 内でローカルアセットを相対パスで参照している場合、これをビルド時に自動検出し、メディアファイルを `/dist/api/assets/` に自動コピーの上、JSON内のパスを公開相対 URL に自動書き換えします。

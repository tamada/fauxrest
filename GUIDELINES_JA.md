# プロフェッショナル開発における暗黙のレギュレーションと心得

本ドキュメントは、チーム開発や実プロダクト開発において、**「あえて開発規約（ルール）に細かく明文化されないが、守ることがプロフェッショナルの大前提とされる」** 重要な作法やベストプラクティスをまとめたものです。

これらは単なる「技術的スキル」ではなく、**「チーム（他の開発者）への敬意」** と **「未来の自分たちへの配慮」** を形にしたものです。これらを身につけ、阿吽の呼吸で実践できるようになることが、一流のソフトウェアエンジニアへの第一歩です。

---

## 🔄 1. バージョン管理（Git / VCS）の作法と歴史の守り方

### 🚨 メインブランチへの「直プッシュ」は絶対禁止
*   どんなに軽微な修正（文字のタイポ修正など）であっても、`main`（または `master`）ブランチへ直接プッシュ、またはローカルで直接コミットしてはいけません。
*   必ずフィーチャーブランチ（作業ブランチ）を切り、GitHub 等の上で **Pull Request (PR) / Merge Request (MR) を作成し、マージによって統合する** というプロセス（統合プロセス）を徹底します。

### 📦 コミットは「アトミック（原子性）」に保つ
*   1つのコミットには「1つの論理的変更」だけを含めます。
*   「新機能の追加」と「無関係な古いコードのクリーンアップ」を1つのコミットに混ぜてはいけません。後からバグが起きたときに、どの変更が原因かを特定できず、`git revert`（コミットの打ち消し）もできなくなります。

### ✍️ コミットメッセージには「なぜ（Why）」を語る
*   「何をしたか（What）」は、コードの差分（`git diff`）を見れば分かります。
*   コミットメッセージの本文には、**「なぜその設計にしたのか」「なぜそのバグが起きたので、どういう設計意図で直したのか」という『理由や背景（Why）』** を詳細に記述します。1行のタイトルだけで済ませるのは、プロの開発では不十分です。

### 🔒 秘匿情報（Secrets）の徹底した排除
*   APIキー、パスワード、秘密鍵、`.env` などのローカル環境設定ファイルをリポジトリにコミットすることはギルティ（重大なセキュリティ違反）です。
*   プロジェクトを開始する際は、コードを書く前にまず `.gitignore` を設定し、安全を確保します。

## 📏 2. コーディングの美学と設計の基本原則

### ✂️ 関数の極小化（1関数最大10行の挑戦）
*   関数やメソッドの長さは可能な限り短く保ちます。1つの関数が20行、30行と肥大化している場合、それは「複数の責任」を1人で抱え込んでしまっているサインです。
*   単一責任（1つのことだけを完璧にこなす）を意識し、徹底的に小さなヘルパー関数へと分割（Decomposition）します。

### ↩️ 早期リターン（Early Return / ガード節）による平坦なコード
*   `if-else` を深くネスト（入れ子に）してはいけません。コードの階層が深くなるほど、脳内での理解負荷（認知負荷）が指数関数的に増大します。
*   条件に合わない不正な値や例外ケースは、関数の冒頭で `if !condition { return; }` のように即座に `return`（ガード節）させ、メインの正常系処理を常に平坦（ネストなし）に保ちます。

### 🧹 ゾンビコード（コメントアウト）の即時物理削除
*   「後で使うかもしれないから」と、古いコードをコメントアウトしたままファイルに残してはいけません。コードがゾンビ化し、現在の仕様がどれなのか他者を混乱させます。
*   過去のコードは Git の歴史からいつでも安全に救出できます。不要になったコードは、容赦なく「物理的に削除」してください。

### 🤖 フォーマッタ（自動整形）の絶対服従
*   スペースの空け方、改行の位置、インデントの深さといった「見た目」の議論に、人間が貴重な時間を費やすべきではありません。
*   プロジェクト開始時に `rustfmt` や `Prettier`、`ESLint` などの静的解析・自動整形ツールを強制導入し、CI（自動テスト）でフォーマット違反があれば即座にビルドエラーにする仕組みを作ります。

## 🧪 3. 品質保証（テスト）に対する不妥協の姿勢

### 🛠️ 「バグ修正」は「テストの追加」と常にセット
*   バグを修正するとき、ただコードを直して「直りました」と報告してはいけません。
*   まず「そのバグを再現する（落ちる）テストコード」を書き、修正コードによってそのテストが「通る（パスする）」ようになることを証明します。テストのないバグ修正は、将来的に同じバグ（デグレーション）を容易に再発させます。

### 📦 テストの「密閉性（独立性）」の確保
*   テストコードの中で「外部の実際のサーバーと通信する」「ローカルマシンの絶対パスを参照する」「マシンの特定のタイムゾーンに依存する」といった不確定要素を入れてはいけません。
*   テストはインターネットが切れたオフライン環境でも、世界中のどの開発者のPCでも、CIサーバーでも、常に100%同じ（決定的）結果を返さなければなりません（モックやスタブの活用）。

### ♻️ 副作用（テストゴミ）の確実なクリーンアップ
*   テストの実行過程で一時的なファイルを作成したり、ローカルDBを書き換えた場合は、テスト終了時に必ずそれらを削除・ロールバックして元の綺麗な状態に戻します。
*   次のテスト実行時や、他の開発者の環境に「テストの残骸」というゴミを残してはいけません（一時フォルダ生成ツールの使用など）。

## 🧼 4. チームを支えるコラボレーションの精神

### 🚦 「クローン直後、1コマンドで動く」ことの価値
*   新しくプロジェクトに入ったメンバーが、リポジトリを `git clone` した後、何時間も環境構築や謎のエラーで迷子になってはいけません。
*   README.md には常に最新かつ正確なセットアップ手順が書かれており、`cargo test` や `npm run dev` などの「1コマンド」を実行すれば、即座に検証可能な状態が保たれているのが、一流のプロジェクトの証です。

### 🗺️ 「暗黙知」をなくすドキュメント化
*   「デプロイ手順は〇〇先輩の頭の中にしかない」「本番環境の環境変数は〇〇さんのチャット履歴から探す必要がある」という状態は、チームにとって致命的なリスク（バス因子）です。
*   あらゆる手順、アーキテクチャ設計、設定項目の仕様は必ず Markdown としてドキュメント化し、全員がいつでも見られる場所にオープンに永続化します。

---

## 🎁 まとめ：コードの綺麗さは「チームへのリスペクト」

優れたソフトウェアは、ただ機能が動くだけでなく、**「次にそのコードを読む人、次にそのコードを保守する人」への思いやり** に満ちています。

これらのレギュレーションを守ることは、一見すると「少し手間が増える」ように思えるかもしれません。しかし、長期的な開発速度、コードのバグ発生率、そしてチームのコラボレーションのしやすさにおいて、手間の何十倍ものリターンをもたらします。

規約で縛られるからやるのではなく、**「プロフェッショナルの誇りと、仲間への思いやり」** として、これらの心得をぜひ手元のコードに宿してください。
